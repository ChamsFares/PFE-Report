\hspace{1.5em}The architecture is guided by four foundational tenets that govern decisions in this chapter.
\begin{enumerate}
    \item the separation of underlay and overlay. The IP underlay provides minimal routed connectivity but does not carry any tenant traffic nor maintain knowledge of logical segments. All tenant mobility and connectivity operations occur at the VXLAN overlay layer, separate from the IP underlay network infrastructure.
    \item symmetry and predictability. Each Leaf node connects to each Spine node, providing symmetric routing with inter-node traffic traversing two hops using ECMP routing. There is no spare link available; the fabric uses its full physical bandwidth capacity for forwarding purposes.
    \item centralized automation with distributed intelligence. The Cisco NDFC operates as one controller, coordinating the management plane for enforcing consistent configurations. However, control plane intelligence operates distributed across the nodes, with independent instances of BGP EVPN and OSPF running on each node. This allows for operation of the fabric independent of the management plane.
    \item defense in depth. Layer 2 segmentation occurs through VNIs, while VRFs segment Layer 3 traffic. Firewall nodes provide perimeter segmentation for tenant traffic.
\end{enumerate}

\section{Platform and Hardware Selection}
\subsection{Hardware Platform Comparison and Selection}
The evaluation of switching hardware options available for the fabric core included four different platforms capable of providing native BGP EVPN-VXLAN capabilities. This assessment considered criteria pertinent to this project specifically, native VTEP capability, maturity of the NX-OS ecosystem, ability to integrate with NDFC, multi-site DCI capability, and adoption by the telecom industry.
table
\hspace{1.5em}As a result, the Cisco Nexus 9000 switch family is chosen for all Spine and Leaf devices because of its native compatibility with Cisco NDFC. Even though alternative devices allow standard BGP EVPN-VXLAN implementation, only the Nexus 9000 family meets all the requirements set forth in Chapter 1. Moreover, it offers ASIC-based VXLAN forwarding that guarantees zero software overhead and, thus, necessary low latency for AI processing.

\hspace{1.5em}The same logic applies to the choice of the Border Leaf devices that remain Huawei CE12800 appliances as they are already present on-site. They are used in the current environment as security and regulatory barriers and are deployed along the site perimeters. The use of such appliances makes a lot of sense since they are able to function as natural Border Gateways for DCI connections thanks to BGP EVPN support.

\hspace{1.5em}Regarding firewalls, there will be two dedicated devices used to ensure the highest level of network security. Thus, Cisco ASAv represents the managed solution responsible for East-West traffic control while being connected directly to the Spine layer. In its turn, FortiGate represents an edge (North-South) unmanaged firewall used for ingress/egress filtering for external communications.

\section{Overlay Design}
In the overlay, VXLAN encapsulation is employed with BGP EVPN being the control plane protocol. A iBGP EVPN setup is utilized where spines operate as Route Reflector. Leafs and border leafs are Route Reflector clients, thereby no full iBGP mesh is required and distribution of control plane is still achieved. Ingress replication is applied for BUM traffic forwarding. No underlay multicast protocols such as PIM and IGMP are needed because of ingress replication which keeps the underlay OSPF only.

\section{Multi-Site Design}
Two sites are connected through a transit AS acting as a simple IP router. Both border leafs at one site connect to their counterparts at another site using the eBGP EVPN configuration with the WAN facing interface IP addresses. Transit AS is responsible for forwarding IP packets (port 179 for BGP TCP connections and UDP 4789 for VXLAN) but is not involved in any EVPN control plane. This architecture makes the WAN layer extremely simple because the transit AS acts only as an IP router; all EVPN smarts reside within the data center fabrics. Direct eBGP connection between border leafs guarantees scalability of both control and data planes and allows operating two EVPN domains independently but consistently.
Border Leafs function as Border Gateways in the NDFC MSD. They republish EVPN routes between different sites – deencapsulating the routes from the fabric and publishing them with site-specific Route Distinguishers. Type-5 IP prefix routes are primarily used for inter-site prefix distribution, which makes tenant subnets available at the other side of the DCI link.
The process of advertising VTEP loopback and the EVPN Type-3 membership route of the on-boarded Leaf happens automatically once the Leaf joins the NDFC POAP workflow. This route is reflected by the local Spine RRs across all existing Leafs. Border Gateways advertise the Type-3 route across the DCI link to the remote site, where it gets reflected across all remote Leafs.

\section{Security Design}

Edge Firewall (FortiGate)  transparent deployment on the border Leafs to filter North-South traffic. Unmanaged Static ACLs.
DC Firewall (Cisco ASAv)  deployed as a service node attached to the two spines. East-West traffic is forced to the ASAv through BGP route leaking. Each tenant VRF has a static default route toward the ASAv’s inside interface. There are also return routes in ASAv back to tenant subnets via spines.
VRF Segmentation  Isolation takes place at Layer 2 and Layer 3 levels, meaning tenants are segmented at Layer 2 and at Layer 3. Cross-VRF communication does not take place unless explicit route leaking is established via matched Route Target import policies. The double-layer isolation approach makes sure that the segmentation persists even if a configuration mistake happens at any layer individually. For instance, VRF-TELECOM is invisible to VRF-WEBHOST unless route-target import/export is intentionally set up.
Management Plane Isolation  Management traffic, including NDFC, SSH, SNMP, syslog, is handled in-band using the dedicated VRF-MGMT, which is a stretched Layer 2 network spanning across both locations by using VLAN 99, L2VNI 10099, and an anycast gateway with the address of 192.168.100.254. This VRF is entirely different from other tenant VRFs. On the physical side, mgmt0 interfaces on switches serve only for emergency out-of-band communication (console recovery).
\section{Automation and Management}

NDFC is the main tool that orchestrates all operations in Cisco Nexus 9000 infrastructure (spines and leaves). It is responsible for lifecycle management of the fabric which includes POAP bootstrapping, Easy Fabric underlay/overlay provisioning, creation of VRF/VNI, compliance verification, and Multi-Site Domain orchestrations. The NDFC tool interacts with Cisco gear in-band VRF-MGMT.

Ansible is responsible for managing border leaf Huawei CE12800 configuration. Ansible playbooks written with huawei.cloudengine collection and NETCONF connections cover all border leafs settings including setting up OSPF neighborships towards Cisco spines, BGP EVPN Border Gateway, interface configuration, and firewall ACLs. Ansible playbooks are put under revision control for traceability.

Day 0 / Day 1 / Day 2 Operations Flow – Day 0: NDFC performs POAP for Cisco underlay/overlay provisioning; Ansible configures border leafs. Day 1: NDFC-guided processes provision VRFs/VNIs, Ansible makes sure that VRF and RT settings on the border leafs match those from NDFC. Day 2: NDFC Compliance Engine validates Cisco nodes' state; Ansible execution might be triggered to bring border leafs back to proper state.
