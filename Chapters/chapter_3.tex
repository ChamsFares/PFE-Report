\section{The Road to Network Automation}
\subsection{CLI Era and Early Monitoring}
\hspace{1.5em}The development of network automation was not sudden; rather, it is the result of decades of continuous improvement. Network complexity, the need for speed in delivering services, and the adoption of SDNs are key factors that have led to network automation.
\\
In the 1990s and early 2000s, most network device management activities were performed using CLI. This meant typing individual commands for individual devices. This approach is error-prone, non-scalable, and manual \cite{Cisco_Catalyst_Programmability_and_Automation}.
Monitoring activities were initially based on SNMP and MIBs. These approaches provided read-only information on metrics such as interface counters and CPU loading averages but provided no native configuration management or programmatic control \cite{IETFRFC3411}.
\subsection{APIs and Programmability}
\hspace{1.5em}This limitation of CLI and SNMP has prompted the advent of programmatic interfaces from the mid-2000s. NETCONF was introduced in RFC 6241, which provided a structured approach to configuration protocols using XML. Later on, RESTCONF was introduced as an alternative to the NETCONF protocol \cite{IETFRFC4741}.
Scripting environments were introduced in network devices. For example, Cisco introduced the Embedded Event Manager, whereas Python was introduced in IOS XE. Around 2011, the advent of Open Flow and SDN has pushed the concept of the separation of control and data planes in the network. The network can be programmed using northbound APIs \cite{OpenFlow_Paper}.
\subsection{Configuration Management and IBN}
\hspace{1.5em}As the API world evolved, the industry began to use existing configuration management tools such as Ansible, Puppet, and Chef, which were originally created to configure servers. These tools were applied to network devices. Ansible was noted for its agentless design and idempotent nature in the form of playbooks\cite{Ansible_Networking_Documentation}.
Intent-Based Networking was introduced in the world of network automation in about 2017-2018. Instead of defining specific network operations, the network operator declares the network intent, such as “Ensure application X has 100 Mbps of connectivity between sites,” and the network automatically figures out the required operations and ensures the network meets the intent\cite{intent-Based_Networking}. The network state is constantly compared with the intended state.
\subsection{The Modern Automation Stack}
\hspace{1.5em}Nowadays, network automation is part of a complete technology stack consisting of:
\begin{itemize}
    \item version control for configurations and automation scripts (Git).
    \item CI/CD pipelines for safe deployment of code changes (Jenkins, GitLab CI).
    \item Infrastructure as Code tools for managing network infrastructure (Terraform, Ansible, NSO).
    \item Observability, including telemetry and streaming analytics instead of traditional SNMP-based polling.
    \item AIOps, which uses machine learning for anomaly detection and root cause analysis \cite{AIOps_and_Machine_Learning}.
\end{itemize}
This technology stack makes it possible to implement continuous delivery, fast rollback, and closed-loop remediation, turning networking into a proactive and software-driven discipline instead of a slow and manual craft.

\section{Automation Architecture}
\subsection{ Northbound and Southbound APIs}
\hspace{1.5em}However, before choosing an automation solution, it is necessary to grasp the conceptual structure of every contemporary network automation framework. 
The API types in controller-based networking are classified based on the direction of their communication flow. 
\begin{itemize}
    \item The northbound API is an interface provided by the network controller for applications and other software above the controller level to communicate with it. third-party software can use these APIs to request network data, deploy policies, and automate processes without the knowledge of the underlying devices or configuration details.
    \item The southbound API is used by the network controller to communicate with the network devices managed by it. The network controller can push configurations, install forwarding tables, get device status, and collect data using southbound APIs.
\end{itemize}

In the approach taken by Cisco, the northbound interfaces used by controllers, like APIC for data center fabric and DNA Center for enterprise fabric, use rich REST APIs, whereas the southbound interfaces leverage protocols such as NETCONF, OpenFlow, or proprietary ones.
\subsection{Orchestration vs Configuration Management}
\hspace{1.5em}While being used synonymously, configuration management and orchestration have completely different purposes in the automation architecture.
\begin{itemize}
    \item Configuration management automates the deployment and configuration of individual systems such as servers, firewalls, routers, or switches. Configuration management makes sure that a particular system is configured according to the desired configuration expressed in code. Automation tools like Ansible and Puppet automate configuration for numerous devices, guaranteeing idempotence (performing the same task multiple times results in the same result) and eliminating manual mistakes. Configuration management works with single or homogenous systems.
    \item turn, orchestration is responsible for coordinating multiple automated actions across multiple systems and services to create a higher-level service. It represents design, delivery, and operation of automated end-to-end services – automating automation itself. The role of an orchestrator is to coordinate several automation tools, run processes against multiple systems, and manage dependencies among them. Provisioning of a three-tiered application might require configuring a load balancer, firewall, virtual machines, and establishing a secure connection via a virtual private network. Orchestration combines these disparate automated actions into a higher-level service.
\end{itemize}

Configuration management and orchestration are not contradictory. An advanced orchestration engine does not negate configuration management software, but rather uses them. The orchestrator invokes the configuration management software to configure individual nodes, along with other functions such as provisioning the infrastructure, updating the DNS, notifying, and rollback.

\subsection{ Declarative vs Imperative Models}
\hspace{1.5em}The last architectural characteristic deals with the representation of automation instructions. There are two entirely opposite paradigms.
\newpage
\begin{itemize}
    \item Imperative paradigm  The administrator provides detailed instructions on the series of commands or API calls to execute in order to reach the desired state. Every instruction is specific: “execute A, then B, and in case of X execute C.” This is the classic CLI-oriented model. It requires extensive syntactical expertise from the administrator along with an understanding of the particular steps needed to be taken in respect of each individual device. As more components are added, imperative processes become lengthy and complicated and must be repeated for each new process
    \item Declarative paradigm  The administrator states the intended outcome without going into details about the sequence of actions. The system is tasked with interpreting the intention and transforming it into appropriate device-specific commands and their execution. For instance, an engineer specifies “VLAN 100 should be configured in all access switches,” and the system figures out what commands will be needed for each component and in what order.
\end{itemize}
\begin{table}[!htbp]
    \centering
    \small
    \renewcommand{\arraystretch}{1.5}
    \begin{tabularx}{\textwidth}{|p{3cm}|X|X|}
        \hline
        \textbf{Aspect} & \textbf{Imperative Model} & \textbf{Declarative Model} \\ \hline
        \textbf{Focus} & \textbf{How} to achieve the state & \textbf{What} the final state should be \\ \hline
        \textbf{Operator Knowledge} & Requires deep syntax and context knowledge for each device & Focuses on describing the desired outcome \\ \hline
        \textbf{Complexity} & Workflow grows with each new component & Intent remains stable; system handles complexity \\ \hline
        \textbf{Idempotency} & Not guaranteed rerunning may cause errors & Built-in; system checks current state before acting \\ \hline

    \end{tabularx}
    \caption{Comparison Between Imperative and Declarative Automation Models}
    \label{tab:imperative_vs_declarative}
\end{table}
From imperative modeling to declarative modeling is one of the trends that dominate the current automation in data centers. While NSO is declarative, model-driven in the Cisco environment, Ansible playbooks, on the other hand, use imperative or even hybrid approaches in some cases.

\section{Virtualization and Emulation}
\subsection{Virtualization}
\hspace{1.5em}Virtualization refers to the process whereby physical resources such as computing, storage, or network resources are mapped into numerous logical resources that operate independently from one another. Server virtualization is the most prevalent form of virtualization in DC networks, where a hypervisor subdivides a physical server into numerous VMs\cite{Virtualization}.

\subsection{Emulation}
\hspace{1.5em}Emulation entails the replication of behavior of a complete computer hardware, comprising its instruction set architecture, memory, input/output and networking functionalities, solely by means of software. In contrast to simulation, which represents an abstraction of the behavior, an emulator operates using the firmware or OS image of the binary code for the real network device. The level of accuracy provided is very high, as the same results will be obtained on the virtual switch.

\subsection{Why Emulation for Network Engineering}
The following are the four reasons why emulation has become necessary for network engineers:
\begin{itemize}
    \item Testing without risk All changes, such as those in configuration, protocols used for routing, or possible failures, can be tested without impacting the live network.
    \item Repeatability The same exact topology and device configurations can be backed up and recreated by any member of the team.
    \item Scalability One physical machine is capable of emulating many virtual routers, switches, and firewalls, making it possible to test a multi-site network fabric prior to acquiring physical hardware.
    \item CI/CD pipeline support The emulated topology can be deployed, tested, and torn down automatically without manual intervention.
\end{itemize}
\subsection{Types of Network Emulators}
Network emulators can be divided into three types depending on their purpose and architecture:
\begin{itemize}
    \item Device emulators emulate a particular device type (such as Cisco vIOS, Nexus 9000v, and Arista vEOS). They are implemented as virtual machines and connected to each other via virtual links.
    \item Topology emulators allow one to create an environment where multiple device emulators can be interconnected using graphs or an API and operate as nodes that are started/stopped and connected/disconnected through links.
    \item Full system emulators emulate complete networks on a packet basis for research purposes (such as Mininet, ns-3).
\end{itemize}
